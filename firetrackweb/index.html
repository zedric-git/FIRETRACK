FireTrack System Docs

Developer documentation for the FireTrack desktop application, including setup, architecture, data flow, repositories, database tables, and user roles.

01. Project Overview

FireTrack is a C# object-oriented desktop application built with WinUI 3 for fire incident management in Cebu City, Philippines.

The project addresses three main problems: there is no centralized system for logging and tracking fire incidents across Cebu City barangays, relief distribution to affected families is difficult to coordinate across agencies, and BFP, DSWD, Barangay officials, and Citizens need one shared platform during an active fire emergency.

FireTrack solves this by giving each operator type (BFP, DSWD, Barangay, Citizen, and Super Admin) its own role-specific dashboard and access controls. BFP logs fires and updates incident status, DSWD records affected families and relief distribution, Barangay users manage evacuation centers, and Citizens can report fires or message DSWD. All roles work from the same MySQL database.

02. Getting Started

Prerequisites

  • Windows 10 or later, build 19041+
  • Visual Studio 2022 with the Windows App SDK workload
  • MySQL 8 running on localhost:3306 with user root and no password
  • .NET 8 SDK

Setup

  1. Clone or open the solution in Visual Studio.
  2. The MySQL database named firetrack is created automatically on first launch. No manual setup is needed.
  3. Build with dotnet build -r win-x64. Do not use plain dotnet build; it will fail.
  4. Run with Ctrl+F5 inside Visual Studio. Do not use F5 because it hangs with WebView2 and multiple windows.

03. Architecture

FireTrack is a single desktop application with no separate backend or API. Everything runs in one process per session.

LayerWhat it does
Views (.xaml + .xaml.cs)UI pages for each role. Code-behind handles page events.
ViewModelsMVVM is used only on the Login and Register pages through CommunityToolkit.Mvvm.
RepositoriesAll database access. Queries use raw SQL through MySqlConnector.
ModelsPlain C# classes that map to database rows.
ServicesSessionManager is static and stores the logged-in user for the current process.
Assets/MapLeaflet.js HTML files loaded inside WebView2.

App Flow

App.xaml.cs -> OnLaunched()
  -> DatabaseHelper.InitializeDatabase()   // creates tables, seeds data
  -> MainWindow                            // hosts a Frame for navigation
       -> LoginPage -> LoadingPage -> [Role Dashboard]

Multi-window behavior: each successful login starts a new background process through LaunchBackgroundLoginWindow(). Each open window has its own isolated SessionManager, so windows do not share session state.

05. How Things Work

Role Routing

After login, App.xaml.cs reads SessionManager.Role and navigates the main frame to the matching dashboard page. BFP goes to BFPDashboardPage, DSWD goes to DSDWDashboardPage, and the other roles follow the same pattern.

WebView2 and C# Bridge

The Leaflet maps run inside WebView2. JavaScript sends map events to C# with window.chrome.webview.postMessage(JSON.stringify({ type: 'pin_dropped', lat, lng })). C# sends data back with webView.CoreWebView2.PostWebMessageAsString(json).

Message types include map_ready, pin_dropped, enable_drop_mode, load_incidents, analysis_mode, brgy_selected, and request_delete_incident.

Account Lockout

After 3 failed login attempts, the account is locked for 15 minutes through LockedUntil in the Users table. On each login attempt, the app checks LockedUntil against the current time. If the lock has expired, it resets FailedAttempts to 0.

Evacuation Occupancy

When a family is assigned to an evacuation center, CurrentOccupancy is increased by MemberCount inside a transaction. If the new total would exceed Capacity, the assignment is blocked and IsFull is set to true. Removing a family decreases the count again.

Repeat Displacement

When a new family is registered, FamilyRepository.IsRepeatDisplaced() checks whether the same HeadName already appears in past incidents. If yes, the new record stores IsRepeatDisplaced as 1.

Idempotent Schema Migration

MigrateEvacSchema() runs every time the app launches and is safe to run again. It checks INFORMATION_SCHEMA before every ALTER TABLE, so it does not duplicate columns or tables.

06. Repositories & Services

Each repository handles one table or domain. All repositories use raw SQL through MySqlConnector; the app does not use an ORM.

RepositoryResponsibility
UserRepositoryAuthenticate with lockout, create users, delete users in a transaction, and hash passwords with SHA-256.
IncidentRepositoryCreate, read, update, and delete incidents; update status; analyze severity by barangay.
FamilyRepositoryRegister families, assign them to evacuation centers, and check repeat displacement.
EvacuationCenterRepositoryCreate, update, and delete centers; update capacity and occupancy; handle cross-barangay center requests.
ReliefRecordRepositoryLog and retrieve relief distributions per family.
BarangayRepositorySeed and retrieve the 80+ Cebu City barangays.
CitizenReportRepositorySubmit and retrieve citizen fire reports, including BFP verification.
DSWDMessageRepositorySend, approve, and reject messages between citizens and DSWD.
AuditLogRepositoryInsert and retrieve timestamped action logs for admin review.
CrossBarangayRequestRepositoryManage requests between barangays for shared evacuation centers.

07. Programming Patterns

Repository Pattern

All database access goes through repository classes. Views do not write SQL directly; they call repository methods. This keeps UI code cleaner and makes the data layer easier to replace or test.

MVVM, Used Partly

Only LoginPage and RegisterPage use full MVVM through CommunityToolkit.Mvvm, including [RelayCommand], [ObservableProperty], and INotifyPropertyChanged. Dashboard pages use code-behind because WebView2 events and multi-panel state were simpler to manage there.

Transactional Deletes

Deleting an incident or user is wrapped in a MySQL transaction. For an incident, the delete process removes relief records, DSWD messages, family evacuation assignments with occupancy rollback, families, and then the incident itself. If any step fails, the whole delete rolls back.

Lazy WebView2 Init

The Citizen dashboard initializes the WebView2 map only when the user first clicks the Map tab. This avoids extra startup time for users who never open the map.

10. User Roles

Super Admin

Full system view. Manages all user accounts, handles lockouts and deactivation, and views the complete audit trail. This account cannot be deleted by any other user.

BFP (Bureau of Fire Protection)

Logs fire incidents with GPS by dropping a pin on the Leaflet map, sets alarm levels 1-3, updates incident status from Active to Under Control to Fire Out, views charts per barangay, and reviews citizen fire reports.

DSWD (Dept. of Social Welfare and Development)

Registers affected families to incidents, manages relief distribution records, tracks DSWD status per incident, and handles citizen messages requesting assistance.

Barangay Ops

Manages local evacuation centers, assigns families to centers, sends cross-barangay requests to use neighboring centers, and views the evacuation map.

Citizen

Browses the incident feed filtered by status, sees which incidents affect their barangay, reports fires, checks relief distribution records, and sends messages to DSWD from the incident detail panel.
